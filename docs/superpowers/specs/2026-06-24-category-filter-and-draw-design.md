# 카테고리 필터 + 2단 추첨 설계

작성일: 2026-06-24

## 목표

추첨에 음식 카테고리(한식/중식/일식/양식/분식/아시아음식/술집 등)를 도입한다.
사용자 입력 부담은 늘리지 않는다 — 사람은 **식당명 + 메뉴**만 입력하고,
카테고리는 카카오에서 자동으로 채운다.

추첨 경로를 두 갈래로 제공한다:
1. **카테고리 직접 선택** → 그 카테고리 식당 리스트 → 식당 추첨
2. **카테고리도 추첨** → 뽑힌 카테고리 식당 리스트 → 식당 추첨

뒤 단계(식당 리스트 → 식당 추첨)는 두 경로가 동일하다.

## 범위

### 포함
- `menus.category` 컬럼 추가
- 카카오 `category_name` 중분류 파싱 → 자동 채움 (신규 등록 시 + 기존 메뉴 앱 시작 시 백필)
- 카테고리 드롭다운(동적: DB에 실제 존재하는 값만)
- "카테고리 추첨" 버튼
- "이 중에서 식당 추첨" 버튼 (기존 추첨을 카테고리 인식하도록 확장)
- 기존 텍스트 검색과 카테고리 필터를 AND로 결합

### 제외 (나중)
- **가격대 필터** — 카카오 API가 가격을 안 줌. 추정은 부정확해서 보류.
- **혼밥/회식 필터** — 카카오에 해당 개념 없음. 카테고리 휴리스틱 추정도 보류.
- 로그인, 한줄평, 단골/블랙리스트 (별도 단계)

## 핵심 제약과 그 함의

1. **카카오 SDK는 브라우저 전용.** 서버(app.js)에서는 카테고리 조회 불가.
   → 카테고리는 **클라이언트가 조회 → 서버에 PATCH로 저장**하는 구조.

2. **추첨은 이미 클라이언트에서 일어난다.** `public/app.js`의 randomBtn 핸들러가
   `/api/menus` 전체를 받아 클라이언트에서 pool을 추리고 뽑는다.
   서버의 `/api/menus/random` 엔드포인트는 **현재 미사용**이며 이번 작업에서 건드리지 않는다.
   → 카테고리 필터도 서버 WHERE가 아니라 **클라이언트 필터**로 넣는다(기존 검색과 동일 방식).

3. **데이터 한 행 = 식당 + 메뉴.** `name`=식당, `description`=메뉴. 식당당 메뉴 한 줄.
   카테고리는 이 식당이 속한 분류이며, 추첨/필터의 그룹 키가 된다.

## 데이터 모델

`menus` 테이블에 컬럼 하나 추가:

```sql
ALTER TABLE menus ADD COLUMN category TEXT;  -- 예: "한식" / "아시아음식" / null
```

- `category_name` 중분류(2번째 토큰)를 그대로 저장. 가공/매핑 없음.
  - `"음식점 > 한식 > 육류,고기"` → `"한식"`
  - `"음식점 > 아시아음식 > 베트남음식"` → `"아시아음식"` (베트남 쌀국수)
  - `"음식점 > 양식 > 멕시칸,브라질"` → `"양식"` (멕시코 타코; 카카오 분류대로)
  - 토큰이 1개뿐이면 그 값을, 검색 실패면 `null`.
- `null` = 아직 미분류(카카오 조회 전 또는 조회 실패).

### initSchema 마이그레이션

`initSchema`는 `CREATE TABLE IF NOT EXISTS`라 기존 테이블엔 컬럼이 안 붙는다.
컬럼 존재 여부를 확인하고 없으면 ALTER 하는 마이그레이션을 추가한다.

```js
// db.js initSchema 내부
const cols = await client.execute("PRAGMA table_info(menus)");
const hasCategory = cols.rows.some((r) => r.name === 'category');
if (!hasCategory) {
  await client.execute("ALTER TABLE menus ADD COLUMN category TEXT");
}
```

## API 변경

### POST /api/menus, PATCH /api/menus/:id
- `category` 필드를 선택적으로 받아 저장/갱신한다. (PATCH는 COALESCE 패턴 유지)
- body에 없으면 기존 동작 그대로(=`null` 유지).

### GET /api/menus
- 기존 그대로 전체 행 반환. `category` 컬럼이 응답에 포함된다.
- 별도 `/api/categories` 엔드포인트는 만들지 않는다. 드롭다운 옵션은
  클라이언트가 받은 목록에서 `category` 값을 추출해 동적 생성한다(아래 UI).

## 클라이언트 (public/)

### map.js — 카테고리 파싱 함수 추가
```js
// "음식점 > 한식 > 육류,고기" → "한식"
export function parseCategory(place) {
  const parts = (place.category_name || '').split(' > ');
  return parts[1] || parts[0] || null;
}
```

### 신규 등록 시 자동 채움
- 식당 추가(POST 성공) 직후, 클라이언트가 식당명으로 `loadKakao()` → `findPlace()`
  → `parseCategory()` → 그 메뉴에 `PATCH { category }`.
- 카카오 키가 없거나(`isLocationEnabled()` false) 조회 실패면 그냥 `null`로 둔다(조용히 스킵).
- 사용자는 식당+메뉴만 입력. 카테고리는 백그라운드에서 채워짐.

### 백필 (앱 시작 시 1회)
- 목록 로드 후 `category == null`인 메뉴를 찾아 순차로 `findPlace` → `PATCH`.
- 카카오 rate limit 고려: 한 번에 하나씩 순차 처리(병렬 금지), 소량씩.
- 카카오 키 없으면 백필 스킵.
- 완료 후 목록 1회 갱신해 드롭다운/리스트에 반영.

### UI 구조
```
[카테고리 영역]
  <select id="category-filter">
    전체 / (DB에 존재하는 category들, null 제외)   ← 직접 선택
  [🎲 카테고리 추첨]                                ← 카테고리도 뽑기
        ↓
[식당 리스트 영역]  (선택/추첨된 카테고리로 필터된 목록)
  [🎲 이 중에서 식당 추첨] → 식당 한 곳 당첨 + 위치 보기
```

### 필터 결합 (검색 AND 카테고리)
- `search.js`의 `filterMenus(menus, query)`를 확장하거나 호출부에서 결합:
  최종 pool = 텍스트 검색 일치 **AND** 카테고리 일치.
- 카테고리 = "전체"면 카테고리 조건 무시.
- 드롭다운 옵션은 현재 검색 결과에 존재하는 category로 동적 생성(빈 옵션 안 띄움).

### 카테고리 추첨
- [카테고리 추첨] 클릭 → 현재 목록(검색 적용 후)에 존재하는 category 집합에서 하나를
  무작위 선택 → 드롭다운을 그 값으로 설정 → 리스트가 그 카테고리로 필터됨.
- 슬롯머신 연출은 기존 식당 추첨과 동일 스타일로 카테고리 이름에 적용(선택).

### 식당 추첨
- 기존 randomBtn 로직 유지. pool이 "검색 + 카테고리"로 필터된 결과가 되도록만 확장.
- pool이 2곳 미만이면 기존처럼 버튼 비활성/안내.

## 엣지 케이스

- **카카오 키 없음**: 카테고리 자동 채움/백필 전부 스킵. 모든 category가 `null`이라
  드롭다운엔 "전체"만, 카테고리 추첨 버튼은 숨김 또는 비활성. 식당 추첨은 기존대로 동작.
- **카테고리 조회 실패/지명 결과**: `null` 유지. 드롭다운에 안 뜨고 "전체"에만 포함.
- **선택한 카테고리에 식당 1곳**: 식당 추첨 비활성(기존 2곳 미만 규칙), 리스트는 표시.
- **선택한 카테고리 + 검색어 조합으로 0곳**: 기존 "검색 결과 없음" UI 재사용.
- **백필 중 사용자가 추첨**: 백필은 비차단(백그라운드). 추첨은 현재 로드된 목록 기준으로 동작.
